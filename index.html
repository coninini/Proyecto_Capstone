Modernización del Sistema de Inventario

Distribuidora de Artículos de Aseo B2B

Propuesta de Proyecto de Título

1. Contexto Actual de la Empresa

  • Operación Principal: Distribuidora en el modelo B2B (empresas comprando a empresas), destacando la atención ágil mediante WhatsApp.
  • Infraestructura Tecnológica: Cuentan con un sistema de ventas básico (tipo POS/Cajero) para el registro de transacciones.
  • Dolor Crítico: El inventario se gestiona externamente en planillas Excel. El flujo de actualización depende del ingreso manual de facturas, lo que genera retrasos y quiebres de stock invisibles.
Tú: Product Owner & QA Tú: Analista e Ing. Datos

2. Capa de Negocio

¿Por qué este proyecto es vital para la empresa hoy?

  • Riesgo en Modelo B2B: Un cliente B2B compra en volumen; fallarle por un Excel desactualizado significa perder rentabilidad mayor.
  • Desconexión Digital: El sistema de ventas actual no "habla" en tiempo real con el inventario físico.
  • Oportunidad Estratégica: La solución implementará gobernanza de datos desde la base, lo que preparará a la empresa para su modernización hacia un E-Commerce B2B automatizado en el futuro.

3. Stack (Backend y BD)

🐘 PostgreSQL (Base de Datos)

- Por qué: Garantiza integridad total (Cumplimiento ACID). El dinero y el stock manejado en inventarios B2B no admite corrupción de datos.

- Cómo: Almacenará relaciones estrictas entre Productos, Usuarios y Transacciones, siendo la única fuente de la verdad.

🟡 Node.js (Servidor API)

- Por qué: Su arquitectura no bloqueante es altísimamente escalable para atender a múltiples vendedores simultáneos.

- Cómo: Actuará como el "cerebro" central (API REST) que recibe peticiones de venta y aprueba/rechaza según parámetros estables.

4. Stack (Frontend y QA)

⚛️ React + TypeScript (Interfaz Web)

- Por qué: React genera fluidez (sin recargar la página) y TS previene bugs humanos al obligar a tipar los datos entrantes estrictamente.

- Cómo: Construiremos un Dashboard (Panel Web) visual e intuitivo donde los gerentes visualizarán el stock y los vendedores registrarán las salidas.

🧪 Selenium (QA Automations)

- Por qué: Validar inventarios a mano tras cada cambio es inviable y peligroso si se omite un bug.

- Cómo: Programaremos scripts automáticos que simularán ser un trabajador revisando que el descuento de stock funcione a la perfección en cada módulo, evitando fallas en Producción.

5. Gobernanza de Datos

Aplicando la teoría a la base de datos para asegurar el éxito:

  • Calidad de Datos (Data Quality): Reglas a nivel de base de datos que impiden anomalías (ej. restricciones de "Stock no puede ser negativo").
  • Fuente Única de Verdad (SSOT): Erradicar la multiplicidad de "Excels". El sistema será el único punto de consulta válido para toda la empresa.
  • Trazabilidad Vital: Cada movimiento u operación debe registrar el usuario, la hora y el documento contable asociado (Auditoría).

6. Seguridad de la Información

  • Control de Acceso (RBAC): Separación estricta de roles. Un cajero realiza ventas; solo un Product Owner/Admin ajusta stock directamente.
  • Protección de Credenciales: Hasheo asimétrico de contraseñas de usuarios.
  • Prevención de Inyecciones: Uso estricto de consultas parametrizadas en la BD para evitar ataques SQL Injection.

7. Marco Legal (Compliance Chile)

  • Ley 21.719 (Delitos Económicos): Un sistema auditable y cerrado previene manipulaciones fraudulentas de inventario empresarial (mitigando riesgos gerenciales).
  • Ley 21.663 (Ciberseguridad Interempresarial): El proyecto adopta bases de seguridad (RBAC) alineadas con la preservación y continuidad operacional de la empresa.
  • Privacidad (Ley 19.628): Resguardo seguro de los datos de contacto y compra de los clientes obtenidos por WhatsApp.
